## 日付

## 画面構成（全4画面）

### 1. ログイン / 新規登録画面
- メールアドレス + パスワードで認証
- 新規登録 / ログインの切り替え
- Supabase Authを利用

### 2. ホーム画面（メイン）
- 天気カード：現在の天気・気温・降水確率・体感温度
- 明日の天気サマリー
- シーン選択チップ（通勤 / デート / カジュアル / アウトドア / フォーマル）
- 「コーデを提案してもらう」ボタン
- 地域表示（設定画面で変更）

### 3. コーデ提案画面（結果表示）
- AIからの提案テキスト
  - トップス / ボトムス / アウター / 小物に分けて表示
  - おすすめカラー（色チップで視覚的に表示）
  - ワンポイントアドバイス
- Unsplashからの参考コーデ画像（2〜3枚）
- 「別のコーデを見る」ボタン（再生成）
- 天気・シーンの表示（何を元に提案したかがわかる）

### 4. 設定画面
- 地域の設定・変更（都道府県 or 都市名で選択）
- ログアウト
- （v2）パーソナルカラー設定

### 画面遷移
```
ログイン → ホーム（天気 + シーン選択）
              └── コーデ提案（結果表示）
              └── 設定（地域変更・ログアウト）
```

### ナビゲーション
- ホームと設定の2画面がメイン。シンプルなヘッダーナビで十分
- コーデ提案画面はホームからの遷移（またはモーダル表示）

## DB設計

### テーブル構成

MVPでは最小限の1テーブルのみ。認証は Supabase Auth が管理。

#### user_settings テーブル
| カラム | 型 | 制約 | 説明 |
|---|---|---|---|
| id | uuid | PK, default gen_random_uuid() | |
| user_id | uuid | FK → auth.users(id), UNIQUE, NOT NULL | |
| region | text | NOT NULL, default '東京' | 地域設定 |
| created_at | timestamptz | NOT NULL, default now() | |
| updated_at | timestamptz | NOT NULL, default now() | |

### MVP段階で「保存しない」もの
- コーデ提案の履歴 → v2でお気に入り機能と一緒に追加
- パーソナルカラー設定 → v2で追加

### RLS（Row Level Security）
- user_settings: `user_id = auth.uid()` で自分の設定のみアクセス可能

## 設計判断のポイント

### DBをほぼ使わない構成にした理由
- MVP機能は「天気取得 → AI提案 → 画像表示」のリアルタイム処理がメイン
- 永続化が必要なのはユーザーの地域設定のみ
- 履歴保存はv2で追加すればよく、MVPではステートレスに保つことで実装を簡素化

### シーン選択を固定にした理由
- MVP段階でカスタムシーン追加のUIは過剰
- 5つの固定シーン（通勤 / デート / カジュアル / アウトドア / フォーマル）で主要ケースはカバーできる
- v2でユーザー定義シーンを検討

### AIプロンプト設計の方針
- 天気情報（気温・天候・降水確率）+ シーンをプロンプトに含める
- 出力はJSON形式で構造化（トップス / ボトムス / アウター / 小物 / カラー / アドバイス）
- Unsplash検索用のキーワードもAIに生成させる
