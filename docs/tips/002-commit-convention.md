# コミットメッセージ規約

## 方針
個人開発のMVPなのでフルのConventional Commitsは導入しない。
ただし、あとで履歴を読み返したときに「何を」「なぜ」変えたかが一目で分かるよう、軽量なプレフィックスだけ統一する。

## 基本フォーマット
```
<prefix>: <日本語で1行サマリ>

（任意）
- 補足の箇条書き
- 実装判断の「なぜ」を1行入れると後で効く
```

- タイトルは日本語、50字以内を目安
- 本文は必要なときだけ。1行で足りるならタイトルだけで終わらせる
- 末尾にピリオド・句点は付けない

## プレフィックス一覧

| prefix | 使いどころ |
|---|---|
| `feat` | 新機能（画面・API・主要コンポーネント） |
| `fix` | バグ修正 |
| `docs` | `docs/` 配下のドキュメントのみ |
| `chore` | 依存追加・設定変更・リネーム・ビルド設定 |
| `refactor` | 挙動を変えない内部整理 |
| `style` | CSS・Tailwind のみの見た目調整 |

迷ったら `chore` でよい。

## 良い例

```
feat: OpenWeatherMap の天気取得 Route Handler を追加

- /api/weather?lat=&lon= で今日と明日の天気を返す
- クライアントにAPIキーを露出させないため Route Handler 経由
```

```
fix: ログイン後のホームへのリダイレクトが効かない問題を修正
```

```
docs: Day 5 のハマりポイントを troubleshooting に追加
```

## 避けたい例

```
various changes     # 何したか分からない
fix bug             # どのバグか分からない
Update page.tsx     # 「何を」「なぜ」が無い
WIP                 # コミット単位で意味を持たせる
```

## Claude Code に依頼するとき
「コミットメッセージは `docs/tips/002-commit-convention.md` に従って」と一言添えるだけで、この規約に沿ったメッセージを付けてくれる。

## 既存コミットへの適用
過去のコミットは**書き直さない**。この規約は新しいコミットから適用する。履歴改変（rebase -i）はメリットが小さく、リモートとの整合を崩すリスクのほうが大きい。
